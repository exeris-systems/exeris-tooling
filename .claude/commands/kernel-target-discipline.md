Audit this change for single-target kernel discipline.

Kernel-target story:
- The 0.1.0 release deliberately removed Spring/Quarkus/Micronaut/Vanilla generators and deleted the multi-backend abstraction. This is intentional.
- The platform commits to a kernel-native runtime story; Spring hosting lives in `exeris-spring-runtime` (a separate repo, NOT generated by this tooling).
- A single backend simplifies emitter shape, parity, determinism, and migration.

Change:
$ARGUMENTS

Please review:
1. Does this change reintroduce backend-specific output (Spring `@RestController`, Quarkus `@Path`, Micronaut `@Controller`, plain `HttpServlet`)?
2. Does it reintroduce a backend-strategy parameter, registry, or feature flag in `exeris-codegen-core`?
3. Does it propose a second `Backend*Generator` or `Spring*Generator` shape?
4. If a downstream wants Spring hosting, is the right answer "use `exeris-spring-runtime` for migration" rather than "generate Spring code here"?
5. Minimal correction if the single-target story is at risk.

A genuine multi-backend reintroduction requires a new ADR (the 0.1.0 single-target story is the decision in force). Do not silently roll it back via a `*Generator` PR.

# ADR-043 — SSE-first kernel HTTP streaming via a sibling `HttpStreamExchange` (link stub)

**Authoritative copy:** [`exeris-kernel/docs/adr/ADR-043-kernel-http-streaming-spi.md`](https://github.com/exeris-systems/exeris-kernel/blob/main/docs/adr/ADR-043-kernel-http-streaming-spi.md)

ADR-043 is a **cross-repo** decision owned by `exeris-kernel`. It introduces a server-push HTTP
streaming primitive in `eu.exeris.kernel.spi.http` — a sibling `HttpStreamExchange` (the streaming
twin of the respond-once `HttpExchange`), an `@FunctionalInterface HttpStreamHandler`, and an
implementation-blind `StreamEvent` record mapping to the SSE wire fields (`event:`/`data:`/`id:`/
`retry:`). Streaming is an opt-in surface the **Core** router selects per-route
(`HttpRouter.Builder.streamRoute(method, path, handler)`); the respond-once invariant is left intact.
Ratifies [RFC-2026-06-18](https://github.com/exeris-systems/exeris-kernel/blob/main/docs/rfc/RFC-2026-06-18-http-streaming-spi.md).

## Why this stub exists — `exeris-tooling` has a concrete obligation

ADR-043 §Context names the missing piece explicitly: the SDK **already advertises** the feature
(`@ExerisDomain(realTimeApi = true)`, `@Action(streaming = true, streamEventType = …, realTimeUpdates)`)
and tooling already parses *part* of it into metadata, but those attributes were **inert because no
kernel primitive existed to bind to**. Kernel 0.10 (`0.10.0-SNAPSHOT`, ADR-043) lands that primitive,
so the SDK → tooling streaming chain is now unblocked. ADR-043 §5 + ADR-005-positive-outcome name the
tooling deliverable: **emit a Java `HttpStreamHandler` + register a streaming route, plus a parity TS
`EventSource`/RxJS client** (Java/TS parity, strong-default #4).

This is **ADR-triggering on the tooling side** (new generator shape + processor↔generator contract +
emitter parity + kernel-target surface) — a tooling ADR/RFC fixes the emitter shape before code lands;
this stub is the awareness anchor, not the design.

## Cross-repo dependency state (validated 2026-06-22; updated 2026-06-24)

- **Kernel** — ✅ ready. `0.10.0-SNAPSHOT` installed locally; the SPI (`HttpStreamHandler`,
  `HttpStreamExchange`, `StreamEvent`, `StreamClosedException`), Core (`SseEventEncoder`,
  `HttpStreamEngine`, `HttpRouter.streamRoute`), and the `AbstractHttpStreamExchangeTck` are present.
- **SDK annotations** — ✅ surface exists (`@Action.streaming/streamEventType/realTimeUpdates`,
  `@ExerisDomain.realTimeApi`).
- **SDK source-model** — ✅ resolved (2026-06-24). `ActionMetadata` was widened with the streaming
  fields (`streaming`/`streamEventType`/`realTimeUpdates`) in the locally-installed SDK, so the
  **per-action** streaming driver's cross-repo blocker (the T1 ↔ `exeris-sdk#58` twin) is cleared;
  what remains for per-action is tooling-side only (processor extraction + the per-action generator
  branch). The **entity-level** `@ExerisDomain.realTimeApi` was already plumbed
  (`DomainMetadata.realTimeApi`, `ExerisDomainProcessor.java:606`) and is what Slice 1 consumes.
- **Tooling** — ✅ Slice 1 shipped (PR #104): `KernelStreamHandlerGenerator` consumes `realTimeApi`
  (no longer inert under `-Aexeris.strict`, T11), `KernelApplicationGenerator` registers
  `streamRoute(GET, {base}/stream, …)`, and `exeris-codegen-ts` emits the parity `EventSource` client.
  The emitter shape is fixed by **ADR-044**. Per-action (Slice 2) is the tracked follow-up.

## When to consult ADR-043 from tooling

- Adding the stream-handler emitter (`Kernel*StreamHandlerGenerator` or an extension of
  `KernelHandlerGenerator`) — the handler binds to `HttpStreamHandler`/`HttpStreamExchange`/`StreamEvent`
  and MUST stay within the SPI (no `text/event-stream` literal, no chunk framing — those are Core's).
- Registering streaming routes in `KernelApplicationGenerator` — use the router's typed
  `streamRoute(method, path, handler)`, **distinct** from the respond-once `route(...)` (obligation 7);
  a streaming route never resolves to an `HttpExchange`.
- The TS parity client (`exeris-codegen-ts`) — `EventSource`/RxJS over the same route; name the parity
  story in the PR (strong-default #4).
- The **event-producer source** — a generated handler must emit `StreamEvent`s from a real producer;
  ADR-043 names `@EventSourced` / `@Projection` as the natural sinks, coupling a *functional* emitter to
  **EV1** (`@DomainEvent` payload realization) / **EV2** (`@EventSourced`, blocked on a kernel
  event-store SPI). A scaffold wired to the domain-event bus is buildable ahead of those.

## Engineering status

ADR-043 is **ACCEPTED** (2026-06-21), kernel side implemented in `0.10.0-SNAPSHOT`. The tooling stream
emitter (ADR-043 §5) has **started and shipped its first slice**: the emitter shape is fixed by
**[ADR-044](ADR-044-tooling-sse-stream-emitter-shape.md)** (ratifies RFC-2026-06-22) and the entity-level
`realTimeApi` live-view landed in **PR #104**. Remaining: (a) ~~tooling emitter-shape ADR/RFC~~ → done
(ADR-044); (b) ~~SDK `ActionMetadata` streaming fields~~ → resolved 2026-06-24, so the **per-action Slice 2**
is tooling-only; (c) the producer source — Slice 1 stands on a deterministic keep-alive scaffold seamed to
the `@DomainEvent` bus, with **EV1** (rich payloads) / **EV2** (`@EventSourced`) realizing it additively.

# ADR-042 — 0.5.0 bidirectional mutation surface + AST-level conflict resolution (link stub)

**Authoritative copy:** [`exeris-sdk/docs/adr/ADR-042-bidirectional-mutation-surface.md`](https://github.com/exeris-systems/exeris-sdk/blob/main/docs/adr/ADR-042-bidirectional-mutation-surface.md)

ADR-042 is a **cross-repo** decision owned by `exeris-sdk`. It defines the 0.5.0 mutation surface (`MutationOp` / `MutationResult` records in `exeris-sdk-source-model`, AST-level three-way conflict detection + application in `exeris-sdk-source-model-io`) and the conflict-resolution model accepted in [RFC-2026-06-11](https://github.com/exeris-systems/exeris-sdk/blob/main/docs/rfc/RFC-2026-06-11-source-model-conflict-resolution.md) (Option B).

## Why this stub exists — `exeris-tooling` has a concrete obligation

Unlike the awareness-only ADR-037 stub, ADR-042 imposes a **real wire-format change on tooling**. The conflict detector in `-io` compares the current source against the last-codegen baseline — and that baseline is the `exeris-metadata/<entity>.json` that **`exeris-tooling` codegen emits**. For the baseline to be trustworthy, ADR-042 obligation #5 requires codegen to emit two new fields into each metadata file:

- **`sourceDigest`** — a content hash of the normalized source the metadata was derived from. The `-io` reader recomputes it against the current source; a mismatch means the baseline is stale and the reader returns `NO_BASELINE`. (The same digest doubles as the 0.5.0 optimistic-concurrency token.)
- **`schemaVersion`** — the `source-model` schema version that wrote the JSON. A reader on a different version refuses (`NO_BASELINE`) rather than compare across schemas.

Without these fields, every `exeris/applyMutation` call degrades to `NO_BASELINE` — the conflict detector has no trustworthy "before" state. This is a **hard sequencing dependency**: the SDK records/detector can land first, but end-to-end bidirectional sync is blocked until codegen emits the two fields.

## When to consult ADR-042 from tooling

- Any change to what `exeris-codegen-*` writes into `exeris-metadata/<entity>.json` — the `sourceDigest` + `schemaVersion` fields are part of the contract now; do not drop or rename them without amending ADR-042.
- Defining the **digest normalization** (line-ending / whitespace neutrality at minimum) — it must match what the `-io` reader recomputes, or every baseline reads as stale. This is a slice-3 coordination point between `exeris-tooling` and `exeris-sdk-source-model-io`.
- Bumping the `source-model` schema in a way that changes the metadata JSON shape — `schemaVersion` must move in lockstep so older baselines are correctly rejected.

## Engineering status

ADR-042 is **ACCEPTED** (2026-06-13). The tooling-side emit change is **ADR-042 slice 3** (cross-repo). **Implemented (0.5.x):** `ExerisDomainProcessor` stamps the two baseline-trust sibling fields (`sourceDigest` + `schemaVersion`) into each `exeris-metadata/<entity>.json` — `sourceDigest` is `SourceDigest.of` over the entity's raw source-file text (the identical input the `-io` reader recomputes), `schemaVersion` is `SchemaVersion.CURRENT`. They are siblings of the serialized `DomainMetadata` (not a wrapper), so a `DomainMetadata` read ignores them and a `BaselineTrust` read of the same file picks up just the two. The metadata JSON is a build intermediate (`target/classes/exeris-metadata`), not committed, so the source-dependent digest creates no committed churn. When the SDK bumps `SchemaVersion.CURRENT`, the stamp follows automatically (symbol reference).

# ADR-049 — Events Log-Ordering & Optimistic-Concurrency Boundary (link stub)

**Authoritative copy:** [`exeris-kernel/docs/adr/ADR-049-events-log-ordering-and-optimistic-concurrency-boundary.md`](https://github.com/exeris-systems/exeris-kernel/blob/main/docs/adr/ADR-049-events-log-ordering-and-optimistic-concurrency-boundary.md)

ADR-049 is a **cross-repo** decision owned by `exeris-kernel`. It turns the previously fire-and-forget
`EventStreamAppender` / `EventStreamReader` seam into the durable-log OCC boundary:
`append(StreamId, long expectedVersion, EventDescriptor, EventPayload)` returns
`AppendResult(committedSequence)` (1-based, strictly monotonic per stream; `ANY_VERSION = -1` opts out)
and fails closed with `EventStreamAppendConflictException` (`EX-EVENT-6008`) on a version conflict;
`EventStreamReader.replayFromVersion` replays in contractually ascending `committedSequence` order.
Kernel 0.10 ships it implemented and bound on **two** Community bindings — JDBC/Postgres
(`exeris_event_log`, per-stream CAS) and Kafka (log-authoritative; OCC is single-writer best-effort,
a documented Community-tier limit) — resolvable via the optional
`KernelProviders.eventStreamAppender()` / `eventStreamReader()` slots.

## Why this stub exists — the EV2 target substrate (no generator consumes it yet)

No tooling generator emits against this surface today — `@EventSourced` is inert and `-Aexeris.strict`
says so honestly (`INERT_ANNOTATIONS`). The stub exists because the ROADMAP **EV2** entry cites
ADR-049 as the delivered **log-level** half of the aggregate-store story, and the EV2 step-2 RFC/ADR
(the codegen-target-shape decision, now actionable) will bind exactly here:

- resolve the `Optional` provider slots; append at `expectedVersion` = current head (0 for a new
  stream); catch `EX-EVENT-6008` and re-read via `replayFromVersion` before retrying;
- what is still **missing at the aggregate level** — an `EventSourcedAggregate` base class, a snapshot
  store, and a typed read surface (`EventStream` yields raw `EventPayload` bytes; decode-side codec
  wiring is deferred per ADR-046) — is the open question the step-2 RFC/ADR must settle (kernel-owned
  vs codegen-emitted load-fold-rehydrate);
- strict-OCC aggregate emission is only safe on the JDBC binding today (the Kafka caveat above) — a
  constraint the emitted shape must not paper over.

When `KernelEventSourcedGenerator` lands (EV2 step 3), the `@EventSourced` entry leaves
`INERT_ANNOTATIONS` in the same change, per the strict-audit rule.

## Engineering status

ADR-049 is **ACCEPTED** (2026-07-01); the decision and implementation slices both shipped on the
kernel v0.10 train (released 0.10.0, 2026-07-02) with the ≥2-binding TCK gate closed (JDBC + Kafka
`EventStream{Appender,Reader}` TCK ITs). Tooling side: no code yet — this stub and the ROADMAP EV2
entry are the record; the next artifact is the EV2 step-2 RFC/ADR.

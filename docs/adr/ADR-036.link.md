# ADR-036 — Server-Side Request Body Decoder SPI (link stub)

**Authoritative copy:** `exeris-kernel` repo → `docs/adr/ADR-036-server-side-request-body-decoder-spi.md` (a sibling repository, not a path within this repo — no working in-repo or GitHub link is possible).

ADR-036 is a **per-repo** decision owned by `exeris-kernel` (kernel/transport scope), with lockstep cross-repo coordination touching this repository for the `KernelHandlerGenerator` request-body decode rewrite (TOOL-139).

## Why this stub exists

`exeris-tooling/exeris-codegen-java/src/main/java/eu/exeris/tooling/codegen/java/kernel/KernelHandlerGenerator.java` previously emitted, into every generated `*Handler`, a static `tools.jackson.databind.ObjectMapper MAPPER` field and a `parseBody` method that copied the request `LoanedBuffer` → `byte[]` → `String` and called `MAPPER.readValue(...)` — a concrete codec symbol baked into application source (a build-time analogue of The Wall, ADR-006) plus a double-allocation on the server ingress path.

ADR-036 closes the fourth HTTP body-codec quadrant (server-side request **decode**) with a real SPI: `HttpRequestBodyDecoder` + `HttpRequestBodyDecoderRegistry` + `HttpRequestDecodingContext`. TOOL-139 rewrites the generator so `parseBody` resolves a decoder from `HttpKernelProviders.httpRequestBodyDecoderRegistry()` and hands it the `LoanedBuffer` directly — no Jackson type, no `byte[]/String` round-trip, in generated output.

## When to consult ADR-036 from tooling

- Any change to `KernelHandlerGenerator.buildParseBody` — the emitted decode path MUST resolve through the SPI registry, never inline a concrete codec. Status mapping stays the handler's job (decode failure → `IllegalArgumentException` → `400`; unregistered decoder → `IllegalStateException` → `5xx`), per ADR-036 §2.
- Any new generator that emits server-side request-decoding code — the same SPI-resolution discipline applies; no `tools.jackson.*` type may appear in generated application source.
- Any E2E generator fixture that **compiles** emitted handler code against the kernel artifact — the fixture needs an `exeris-kernel-spi` snapshot that already publishes `HttpRequestBodyDecoder` / `HttpRequestBodyDecoderRegistry` / `HttpRequestDecodingContext` (the kernel-side HTTP-138 change). The string-match unit tests (`KernelHandlerGeneratorTest`) do not need the snapshot; the compile-gate does — so TOOL-139 lands in lockstep, in a subsequent kernel-snapshot consumption cycle (cf. the ADR-034 / TOOL-136 precedent).

## Engineering status

Kernel half (HTTP-138: SPI + `CommunityJsonRequestBodyDecoder` + wiring + TCK) lands in `exeris-kernel`. This repository's generator rewrite (TOOL-139) is the lockstep companion; it does not block the kernel merge.

# ADR-037 — Isolate the source-model parser/writer in `exeris-sdk-source-model-io` (link stub)

**Authoritative copy:** [`exeris-sdk/docs/adr/ADR-037-source-model-io-module.md`](../../../exeris-sdk/docs/adr/ADR-037-source-model-io-module.md)

ADR-037 is a **cross-repo** decision owned by `exeris-sdk`. It places the 0.3.0 `.java` ↔ `DomainMetadata` parser/writer (JavaParser-based) in a new sibling module `exeris-sdk-source-model-io`, keeping `exeris-sdk-source-model` dependency-light so pure AST-record consumers preserve zero runtime coupling. Driven by [RFC-2026-06-03](../../../exeris-sdk/docs/rfc/RFC-2026-06-03-source-model-parser-writer.md).

## Why this stub exists

`exeris-tooling` consumes the SDK's AST **records** — `exeris-processor` builds `DomainMetadata` from JSR-269 (`javax.lang.model`) during `javac` and serializes them; `exeris-codegen-*` reads those records to emit Java/TS/OpenAPI. Neither parses or rewrites `.java` source, so **neither should ever depend on `exeris-sdk-source-model-io` or transitively pull JavaParser.** ADR-037 obligation #2 makes JavaParser exclusive to `-io`; this stub records the tooling-side half of that boundary so a future contributor doesn't "simplify" by routing codegen through the parser module.

## When to consult ADR-037 from tooling

- Any proposal to add `.java` source parsing to a processor or generator — do it via `exeris-sdk-source-model-io`, and keep that dependency out of artifacts that only need the AST records (the processor's JSON layer, the codegen emitters).
- Any dependency change that would put `com.github.javaparser:*` on the `exeris-processor` or `exeris-codegen-*` dependency tree — it violates ADR-037 #2. The codegen path emits *fresh* files and never needs JavaParser; the idempotent writer is `-io`'s job, not codegen's.
- Any new generator that round-trips user-authored source (rewrites in place rather than emitting fresh) — that is writer territory; the canonical writer lives in `-io`, so reuse it rather than re-implementing in tooling.

## Engineering status

ADR-037 is **ACCEPTED** (2026-06-03). No tooling code change ships with it — `exeris-tooling` stays on `exeris-sdk-source-model` for the AST records. This stub is awareness-only: it exists so codegen/processor authors find the zero-coupling boundary without searching the SDK repo.

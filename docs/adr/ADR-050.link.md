# ADR-050 — Events Binding-Agnostic `topic` (link stub)

**Authoritative copy:** [`exeris-kernel/docs/adr/ADR-050-events-binding-agnostic-topic.md`](https://github.com/exeris-systems/exeris-kernel/blob/main/docs/adr/ADR-050-events-binding-agnostic-topic.md)

ADR-050 is a **cross-repo** decision owned by `exeris-kernel`. It gives the SDK `@DomainEvent.topic`
attribute a kernel sink: `topic` becomes an optional, binding-agnostic `String` on `EventTypeSpec` — the
per-event-*type* registration record — **not** a field on the per-instance, primitive-only,
Valhalla-ready `EventDescriptor`. Bindings resolve it through the `EventRegistry`; the Kafka binding maps
it to the concrete broker topic on both publish and subscribe (an override of the default type-name
topic); the in-memory `EventBus` treats it as advisory (topic-blind by design, ADR-049 H2).

## Why this stub exists — `exeris-tooling` has a concrete obligation

Before ADR-050 the kernel offered no sink for `topic`, so `KernelEventGenerator` **dropped the captured
value to a Javadoc-only reference** ("preserved here only for reference"). The kernel was the blocker,
not the tooling. ADR-050 closes it. The tooling lockstep (ADR-050 Engineering Protocol step 2):

- **`KernelEventGenerator` populate** — for an event whose `DomainEventMetadata.topic` is non-blank, the
  emitted `*EventPublisher` builds its `EventTypeSpec` via the three-arg
  `EventTypeSpec.ofPersistent(name, ordinal, topic)`; an event without a topic keeps the two-arg factory
  (`topic = null` → "no override"). Redaction / codec resolution (ADR-046) are unchanged.
- **The Wall** — the generated publisher still names only SPI symbols; `topic` is a plain SPI `String`
  and broker meaning stays binding-private (no `org.apache.kafka.*` leak).
- **e2e** — `KernelCodegenE2ETest` asserts a captured `@DomainEvent.topic` lands on the spec (three-arg
  factory) while a no-topic event stays two-arg; the `KernelCodegenCompileTest` fixture carries a
  topic-bearing `@DomainEvent`, so the gate `javac`-compiles the three-arg `ofPersistent` against the
  live kernel SPI.

The `exeris-sdk` side updates the `@DomainEvent.topic` "Open-Core status" Javadoc to the new stance
(carried on `EventTypeSpec`; broker bindings honour it, the in-memory bus is advisory).

# ADR-045 — `HttpRetryPolicy` SPI: Opt-in Client-Side Retry (link stub)

**Authoritative copy:** [`exeris-kernel/docs/adr/ADR-045-client-side-http-retry-policy-spi.md`](https://github.com/exeris-systems/exeris-kernel/blob/main/docs/adr/ADR-045-client-side-http-retry-policy-spi.md)

ADR-045 is a **cross-repo** decision owned by `exeris-kernel`. It resolves the retry deferral of
ADR-026/ADR-032: a small implementation-blind SPI `HttpRetryPolicy` in `eu.exeris.kernel.spi.http`
(`RetryDecision decide(HttpRequest, HttpAttemptOutcome, int attemptIndex)`; `none()` preserves the
no-implicit-retry default), consumed by `KernelWebClient` through one new **opt-in constructor
parameter** — the same composition shape ADR-032 established for the enricher. Behavioural defaults
live in the Community impl `CommunityHttpRetryPolicy` (idempotency gate per RFC 9110 +
`Idempotency-Key` opt-in, `502/503/504` + transport failures, `Retry-After` honouring, capped
exponential backoff with full jitter, 3-attempt cap) and can be tuned without re-opening the ADR.

## Why this stub exists — `exeris-tooling` has a concrete obligation

`KernelClientGenerator` was one of the three consumers that forced the seam: a transient 503 on a
generated `*Client` call surfaced as a `WebClientException` on the first attempt, with nowhere to
express retry that the generator would not round-trip away. The tooling lockstep (ADR-045
Implementation plan, Cross-repo):

- **Composition-root example, no per-entity codegen change** — the policy is constructor-injected
  into the shared `KernelWebClient` like the enricher, so one wiring gives every emitted client
  retry semantics. The emitted `*Client` constructor Javadoc now carries the example:
  `new KernelWebClient(engine, allocator, requestEncoders, responseDecoders,
  HttpClientRequestEnricher.noop(), new eu.exeris.kernel.community.http.CommunityHttpRetryPolicy())`,
  with `HttpRetryPolicy.none()` named as the no-implicit-retry opt-out (ADR-026).
- **The Wall** — the example is Javadoc text only (plain FQN, never a JavaPoet `$T`), so no
  `eu.exeris.kernel.community.*` import lands in the compiled surface; generated application source
  keeps naming only the tier-neutral `KernelWebClient` facade (ADR-034).
- **Tests** — `KernelClientGeneratorTest` pins the example and the Wall (`doesNotContain` on the
  Community import); `KernelCodegenCompileTest` is unaffected by design — Javadoc does not resolve
  at `javac` time, so the stub classpath still needs only `KernelWebClient`.

The generated client's request surface, verbs, and `404 → Optional.empty()` mapping are unchanged;
retry remains invisible to per-entity code by construction.

# ADR-038 — SDK realization of the capability annotation surface (link stub)

**Authoritative copy:** [`exeris-sdk/docs/adr/ADR-038-capability-annotation-surface.md`](https://github.com/exeris-systems/exeris-sdk/blob/main/docs/adr/ADR-038-capability-annotation-surface.md)

ADR-038 is a **cross-repo** decision owned by `exeris-sdk`. It is the SDK-side realization of [ADR-024](https://github.com/exeris-systems/exeris-docs/blob/main/adr/ADR-024-capability-composition-model.md)'s capability-composition model: `@CapabilityModule` / `@Provides` / `@Requires` / `@CapabilityLifecycle` are defined in `exeris-sdk-annotations` (`eu.exeris.sdk.annotation.capability`, `@Retention(SOURCE)`), services are referenced by `Class<?>` and persisted in the AST as source-written name strings (tooling-normalized to FQN), and `CapabilityModuleMetadata` / `ProvidesMetadata` / `RequiresMetadata` records live in `exeris-sdk-source-model`. Driven by [RFC-2026-06-03](https://github.com/exeris-systems/exeris-sdk/blob/main/docs/rfc/RFC-2026-06-03-capability-annotation-surface.md).

## Why this stub exists

`exeris-tooling` is the **consumer** of this surface, on both halves of ADR-024's build-time pipeline:

- **`exeris-processor`** reads the capability annotations via JSR-269 (`javax.lang.model`) — the same way it reads `@ExerisDomain` — and emits the `Capability*Metadata` AST records. It must consume the SDK annotation types, **not redefine them**: the annotation surface is the SDK's, exactly as for the entity surface.
- **`exeris-codegen-*`** reads those AST records to do the work ADR-024 assigns to tooling: `@Requires`→`@Provides` resolution, the dependency DAG, version-range intersection, the cap-tier Wall checks, and emitting `cap-manifest.json`. ADR-038 explicitly keeps all of that **out of the SDK** — the SDK ships only the annotations + records.

This stub records the tooling-side half of the boundary so a future contributor doesn't add a competing capability annotation in tooling, or push resolution/manifest logic back up into the SDK.

## When to consult ADR-038 from tooling

- Adding capability processing to `exeris-processor` — read the SDK annotations and build `CapabilityModuleMetadata`/`ProvidesMetadata`/`RequiresMetadata`; mirror the SDK record shapes (service = source-written name string, which **tooling** normalizes to FQN; `@Provides`/`@Requires` flatten their nested `.List` containers).
- Service identity normalization — the SDK's `-io` reader matches by simple name (no symbol solving), so the AST may carry a written (possibly simple) name; **FQN normalization is tooling's job** (ADR-038 obligation #6). The JSR-269 processor already yields the FQN; keep the two paths reconciled.
- Anything resolution/DAG/version/Wall/manifest-shaped — that is tooling scope per ADR-024 + ADR-038, not a candidate for an SDK annotation field.

## Engineering status

ADR-038 is **ACCEPTED** (2026-06-05). No tooling code change ships with it — the annotation surface + AST records land in `exeris-sdk` first (annotations → records → `-io` reader slices). This stub is awareness-only: the capability processing in `exeris-processor` / `exeris-codegen-*` is the downstream consumer that lands once the SDK surface exists.
